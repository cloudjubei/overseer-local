# Agent Runner Service: Overview and API

This document specifies the design for the Agent Runner Service, a dedicated micro-service responsible for executing agent-based tasks in a controlled, scalable, and observable manner.

## 1. Overview

The Agent Runner Service is a core component of the backend architecture. Its primary responsibility is to manage the lifecycle of "agent jobs." An agent job represents the execution of a specific agent (e.g., "Developer Agent") on a given task within a Git repository.

The service operates on a job queue, ensuring that tasks are processed reliably. It interacts with Git repositories to fetch the context for a task and to commit the results of the agent's work.

Key Responsibilities:
- Manage a queue of agent jobs.
- Execute agents using a shared library (derived from `factory-ts`).
- Handle the entire lifecycle of a job, from queuing to completion or failure.
- Interact with Git repositories to check out code and commit results.
- Provide detailed logging and status updates for each job.
- Handle artifacts generated by agents and commit them back to the repository.

## 2. Architecture

The Agent Runner is designed as a standalone micro-service. It communicates with the main backend service, which acts as the primary interface for clients.

- **Communication**: The main backend and the Agent Runner communicate via a message queue (e.g., RabbitMQ or Redis Streams) for job submission and event notifications. This provides a robust, asynchronous, and scalable integration pattern. An auxiliary REST API on the Agent Runner can be used for direct queries (e.g., getting job status).

- **Core Logic**: The service will use a shared library, refactored from the existing `factory-ts` package, to orchestrate agent runs. This ensures that agents behave consistently whether run locally by a client or remotely via the backend.

- **Environment**: Each job is executed in an isolated environment (e.g., a temporary directory or a container) with a fresh clone of the specified Git repository.

## 3. Job Lifecycle

A job progresses through a well-defined set of states:

1.  **`queued`**: The job has been received and is waiting for a worker to become available.
2.  **`running`**: A worker has picked up the job and is executing the agent. The service will clone the repository, install dependencies, and start the agent.
3.  **`completed`**: The agent finished its work successfully. Any file changes have been committed to the repository.
4.  **`failed`**: The agent execution failed due to an error. The error details are logged and reported.
5.  **`cancelled`**: The job was cancelled by a user request before it could be completed.

## 4. Job Queue and Concurrency

- **Queue**: Jobs are submitted by the main backend to a dedicated message queue topic (e.g., `agent.jobs.new`).
- **Workers**: The Agent Runner service will run a pool of workers that listen to this queue.
- **Concurrency**: The number of concurrent jobs is determined by the number of active workers. This can be configured and scaled horizontally by deploying more instances of the Agent Runner service.
- **Prioritization**: (Future) The queue can be configured to support different priority levels for jobs.

## 5. API and Event Contracts

Communication is primarily event-driven, with an optional REST API for synchronous queries.

### 5.1. Job Submission (Message Queue)

The main backend enqueues a job by sending a message to the `agent.jobs.new` topic.

**Message Payload:**
```json
{
  "jobId": "uuid-v4-string",
  "repository": {
    "url": "https://github.com/user/repo.git",
    "branch": "main",
    "commit": "optional-specific-commit-sha"
  },
  "agent": {
    "id": "DeveloperAgent",
    "version": "1.0.0"
  },
  "task": {
    "id": "task-uuid",
    "description": "Implement the login feature."
  },
  "feature": {
    "id": "feature-uuid",
    "description": "As a user, I want to log in."
  },
  "metadata": {
    "userId": "user-uuid",
    "timestamp": "iso-8601-timestamp"
  }
}
```

### 5.2. Event Notifications (Message Queue)

The Agent Runner publishes events to a notification topic (e.g., `agent.jobs.events`) that the main backend subscribes to.

**Event: `job.status.changed`**
This event is sent for every state transition in the job lifecycle.

**Payload:**
```json
{
  "jobId": "uuid-v4-string",
  "status": "queued|running|completed|failed|cancelled",
  "timestamp": "iso-8601-timestamp",
  "data": {
    // Optional data depending on status
    "commit": "new-commit-sha-on-completion", // for 'completed'
    "error": "Error message and stack trace", // for 'failed'
    "reason": "Cancelled by user" // for 'cancelled'
  }
}
```

**Event: `job.log.created`**
This event is for streaming agent logs/output back to the main backend.

**Payload:**
```json
{
  "jobId": "uuid-v4-string",
  "timestamp": "iso-8601-timestamp",
  "stream": "stdout|stderr",
  "message": "Log line from the agent."
}
```

### 5.3. REST API (Optional)

The Agent Runner can expose a simple REST API for direct status checks.

- `GET /jobs/{jobId}`
  - **Description**: Get the current status and history of a job.
  - **Response**:
    ```json
    {
      "jobId": "uuid-v4-string",
      "status": "running",
      "history": [
        { "status": "queued", "timestamp": "..." },
        { "status": "running", "timestamp": "..." }
      ],
      "agent": { "id": "DeveloperAgent" },
      "task": { "id": "task-uuid" }
    }
    ```

- `POST /jobs/{jobId}/cancel`
  - **Description**: Request cancellation of a job.
  - **Response**: `202 Accepted`

## 6. Artifact Handling

1.  **Clone**: For each job, the service performs a fresh clone of the specified repository and branch.
2.  **Execution**: The agent runs within the context of this cloned repository.
3.  **Commit**: Upon successful completion, the agent is expected to have staged its changes using `git add`. The service is responsible for creating a commit with a standardized message.
    - **Commit Message Format**: `feat(agent): Implement feature 'feature-uuid' for task 'task-uuid'\n\nAgent: DeveloperAgent\nJob: 'job-uuid'`
4.  **Push**: The service pushes the new commit(s) to the repository's branch.
5.  **Reporting**: The final commit SHA is reported back in the `job.status.changed` event for the `completed` state.

## 7. Integration with `factory-ts` (Shared Library)

- The core agent orchestration logic from `factory-ts` will be extracted into a shared library, consumable as an NPM package.
- The Agent Runner service will be a primary consumer of this library.
- The library will be responsible for:
  - Instantiating the correct agent.
  - Managing the agent's interaction with the filesystem and external tools (LLMs).
  - Providing hooks for logging and progress reporting that the Agent Runner will use to publish events.

This design ensures that agent logic is decoupled from the execution environment (local client vs. cloud service) and can be reused consistently.
